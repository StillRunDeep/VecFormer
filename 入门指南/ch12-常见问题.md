## 环境配置问题

### Q1：`import flash_attn` 报错 / Flash Attention 安装失败

**症状**：
```
ImportError: No module named 'flash_attn'
# 或
RuntimeError: CUDA error: no kernel image is available for execution on the device
```

**原因**：flash_attn 未安装，或编译时未包含当前 GPU 的 CUDA 架构（sm_xx）。

**解决方案**：

```bash
# 方案一：指定 GPU 架构从源码编译（推荐）
pip install packaging ninja
git clone https://github.com/Dao-AILab/flash-attention.git
cd flash-attention

# 根据你的 GPU 设置架构：
# RTX 2080 Ti → sm_75
# RTX 3090/3080 → sm_86
# A100 → sm_80
# RTX 4090 → sm_89
CUDA_ARCH=75 MAX_JOBS=8 python setup.py install
cd ..
```

```bash
# 方案二：不安装，使用自动 fallback（功能正常，速度略慢）
# 无需任何操作，模型已修复：flash_attn 不可用时自动使用 SDPA
# 启动时会看到 UserWarning，属于正常现象
```

### Q2：`spconv` 报 CUDA 相关错误

**症状**：
```
RuntimeError: CUDA error when calling spconv operations
```

**原因**：`spconv-cu118==2.3.8` 预编译版本可能不包含当前 GPU 架构的 kernel。

**解决方案**：

```bash

python -c "
import torch, spconv.pytorch as spconv
print('spconv OK, version:', spconv.__version__)
x = spconv.SparseConvTensor(
    torch.ones(10, 32).cuda(),
    torch.zeros(10, 4, dtype=torch.int32).cuda(),
    [10, 10, 10], 1
)
print('spconv CUDA test OK')
"

# 如果失败，从源码编译
pip uninstall spconv-cu118
TORCH_CUDA_ARCH_LIST="7.5" pip install spconv --no-binary spconv
```

### Q3：`torch_scatter` 安装后仍报 ImportError

**症状**：
```
ImportError: cannot import name 'scatter' from 'torch_scatter'
```

**原因**：torch-scatter 版本与 PyTorch 版本不匹配（ABI 不兼容）。

**解决方案**：

```bash
# 确保 PyTorch 版本正确
python -c "import torch; print(torch.__version__)"
# 应该是 2.5.1+cu118

# 重新安装 torch-scatter（使用正确版本号）
pip uninstall torch-scatter
pip install torch-scatter -f https://data.pyg.org/whl/torch-2.5.1+cu118.html
```

## 运行时错误

### Q4：CUDA out of memory（显存不足）

**症状**：
```
RuntimeError: CUDA out of memory. Tried to allocate XX GiB
```

**常见原因与解决**：

```yaml
# 方案一：减小 batch size（configs/vecformer.yaml）
per_device_train_batch_size: 1   # 从 2 降到 1
per_device_eval_batch_size: 1

# 方案二：减小 patch_size（减少每次注意力计算量）
# configs/model/vecformer.yaml 中的 model_args 或 configuration_vecformer.py
backbone_config:
    enc_patch_size: [512, 512, 512, 512, 512]   # 从 1024 改为 512

# 方案三：减少 num_topk_preds（推理时内存）
num_topk_preds: 300   # 从 600 降到 300

# 方案四：确认没有梯度泄漏（推理代码中加 no_grad）
with torch.no_grad():
    output = model(batch)
```

### Q5：训练时 loss 变成 NaN

**症状**：训练几步后 loss 输出 `nan`

**常见原因**：

```
1. 学习率太大 → 梯度爆炸
   解决：降低 learning_rate（如从 1e-4 降到 5e-5）

2. 批次中有全为某一类别的样本 → Dice loss 计算不稳定
   解决：检查数据预处理是否正确

3. fp16 溢出（如果使用混合精度）
   解决：检查 upcast_attention=True / upcast_softmax=True 是否开启

4. enable_flash=True 但 flash_attn 未正确安装
   解决：设置 enable_flash=False，或重新编译 flash_attn
```

**调试方法**：

```python
# 在损失计算后加检查
assert not torch.isnan(loss), f"Loss is NaN at step {step}"
assert not torch.isinf(loss), f"Loss is Inf at step {step}"
```

### Q6：AssertionError: Set upcast_attention to False when enable Flash Attention

**症状**：

```
AssertionError: Set upcast_attention to False when enable Flash Attention
```

**原因**：`enable_flash=True` 与 `upcast_attention=True` 互斥。Flash Attention 内部已处理精度问题，不需要 upcast。

**解决方案**：

```python
# 二选一：
# 选项A：使用 Flash Attention（需要正确安装）
backbone_config = dict(
    enable_flash=True,
    upcast_attention=False,   # 必须为 False
    upcast_softmax=False,     # 必须为 False
)

# 选项B：不使用 Flash Attention（兼容所有 GPU）
backbone_config = dict(
    enable_flash=False,
    upcast_attention=True,    # 提高数值稳定性
    upcast_softmax=True,
)
```

## 性能问题

### Q7：训练速度很慢

**诊断步骤**：

```bash
# 1. 检查 GPU 利用率
watch -n 1 nvidia-smi

# 预期：GPU 利用率 > 80%
# 如果低：可能是 CPU/IO 成为瓶颈

# 2. 检查数据加载是否够快
# configs/vecformer.yaml
dataloader_num_workers: 64   # 根据 CPU 核心数调整（通常设为 4 × GPU 数）
```

**常见瓶颈**：

| 问题 | 现象 | 解决 |
|------|------|------|
| flash_attn 未安装 | 注意力计算慢 3–5 倍 | 安装 flash_attn |
| 数据加载慢 | GPU 利用率低，数据加载时间长 | 增大 `num_workers` |
| 单 GPU 训练 | 吞吐量低 | 使用 `torchrun --nproc_per_node=N` |
| `upcast_attention=True` | 注意力计算用 float32，慢 | 安装 flash_attn 后改为 `enable_flash=True` |

### Q8：评估 PQ 指标很低（< 0.1）

**可能原因**：

```
1. 训练轮数不够
   → 模型需要 ~200 epoch 才开始收敛，500 epoch 达到最佳
   → 检查 eval/PQ 的 TensorBoard 曲线是否还在上升

2. 数据预处理有问题
   → 检查 datasets/FloorPlanCAD-sampled-as-line-jsons/val/ 是否有数据
   → 验证：python -c "from data.floorplancad import FloorPlanCAD; d=FloorPlanCAD(...,'val',...); print(len(d))"

3. 学习率配置问题
   → 检查 TensorBoard 中 train/learning_rate 曲线是否符合 warmup_stable_decay 形状

4. batch_size 太小（单 GPU 训练）
   → 总 batch_size 应 ≥ 8，单 GPU 时增加 gradient_accumulation_steps

5. 加载了错误的 checkpoint
   → 确认 resume_from_checkpoint 路径正确
```

## 代码理解问题

### Q9：`cu_seqlens` 和 `offset` 有什么区别？

```
cu_seqlens：本项目定义，整个代码库通用
  → int32 张量，dtype 固定
  → 用于所有 VarlenAttention 函数

offset：PTv3 内部用的等价概念
  → int64 张量
  → 通过 offset2batch() / offset2bincount() 转换

它们的值相同（都是累计序列长度），只是 dtype 和命名约定不同。
```

### Q10：`prim_ids` 和 `layer_ids` 有什么区别？

```
prim_ids（图元 ID）：
  - 同一个建筑符号（如一扇门）的所有线段共享相同的 prim_id
  - 同一张图内，不同符号有不同 prim_id
  - 用于 GroupFeatFusion：按图元聚合特征

layer_ids（图层 ID）：
  - CAD 软件中，建筑师把相同类型的线段放在同一"图层"
  - 例：所有门的线段在"门图层"（layer_id=5）
  - 同一图层的线段有更强的语义相关性
  - 用于 FusionLayerFeatsModule：按图层聚合特征
```

### Q11：为什么 `pred_inst_masks` 的形状是 `(Q, N)` 而不是更小？

```
直觉：有 Q 个查询向量，N 条线段，每个查询要预测"哪些线段属于我"

计算方式：
  masks = queries @ backbone_feats.T
  (Q, 256) @ (256, N) = (Q, N)

这是一个内积矩阵：
  masks[i, j] 越大 → 查询 i 和线段 j 越"相似" → 线段 j 越可能属于实例 i

Q=N=800 时：(800, 800) = 640000 个标量，对于短序列是合理的
如果 N 很大（>10000），可以考虑减小 num_topk_preds 或使用稀疏掩码
```

### Q12：`iter_pred=True` 时，6个 Block 的损失权重一样吗？
